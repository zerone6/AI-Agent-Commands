PLAN 파일에 따라 구현을 수행한다.

## 실행 순서

### 0. 사전 항목 확인

- [ ] `git status` 클린 상태 확인
- [ ] feature 브랜치 생성 및 체크아웃
- [ ] PLAN 파일이 사용자 승인 완료 상태인지 확인

### 1. PLAN 파일 선택

`/docs/current_sprint/`에서 최신 PLAN 파일을 선택한다.

- 정렬 기준: MMDD 내림차순 → NN 내림차순
- 예: plan122403.md > plan122402.md > plan122301.md

### 2. 실행 모드 결정

| 모드       | 조건               | 동작                                                     |
| ---------- | ------------------ | -------------------------------------------------------- |
| 기본       | 특별 지시 없음     | 수동 테스트 필요 시점까지 진행 후 사용자에게 테스트 요청 |
| 특정 Phase | "Phase N까지" 지시 | 해당 Phase까지만 진행                                    |
| 전체       | "전체 구현" 지시   | 모든 Phase 완료                                          |

### 3. Phase별 구현

각 Phase마다:

1. 구현 수행
2. 자동화 테스트 실행 (API 테스트 등 해당 시)
3. PLAN 파일 체크리스트 업데이트
4. 수동 테스트 필요 시 → 사용자에게 요청 후 대기

### 4. 실패 시 행동

| 상황            | 행동                             |
| --------------- | -------------------------------- |
| 테스트 실패     | 즉시 중단, 원인 분석 후 보고     |
| 구현 중 에러    | 중단, 에러 보고 형식에 따라 보고 |
| 요구사항 불명확 | 중단, 사용자에게 질문            |

### 5. 완료 처리

모든 구현 완료 시:

1. AI 자동 테스트 실행
   - AI Agetn에 의한 자동테스트 계획 작성
   - JUnit 테스트 등 자동화 테스트 계획을 작성하고, 사용자에게 확인을 받음
   - 자동화 테스트 실행 및 결과 보고
2. PLAN 파일에 결과 기록
   - 리스크 및 롤백: 발생한 이슈
   - 자동화 테스트 결과 기록
     - 자동화 테스트 실행 리스트 일람, 성공항목, 실패 항목 기록, 실패 항목의 원인과 대책 기입
   - 수동 테스트 항목 작성
     - 사용자가 실행해야할 수동 테스트 항목 작성
     - 사용자가 수동 테스트를 완료한 후, PLAN 파일에 결과 기록
   - TODO: 추가 작업 항목
3. 사용자에게 완료 보고
4. 커밋은 전체 완료 후 1회 수행 (finish_order 커맨드에서 처리)

> 이후 finish_order 커맨드를 통해 커밋 및 브랜치 정리를 진행한다.
