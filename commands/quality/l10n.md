화면에 표시되는 모든 텍스트에 다국어 지원을 적용하고 누락 항목을 검증합니다.

## 목적

사용자 대면 텍스트를 하드코딩에서 번역 키로 전환하여 다국어 지원을 가능하게 하고, 언어별 번역 누락을 방지합니다.

## 트리거

- 신규 기능 개발 후 l10n 누락 확인이 필요할 때
- 릴리즈 전 l10n 전체 점검이 필요할 때
- 사용자가 l10n 작업을 요청할 때
- 하드코딩된 텍스트가 발견되었을 때

## 결과물

### 파일 업데이트

PLAN 파일이 존재하는 경우 해당 PLAN에 l10n 섹션 추가:

```markdown
## l10n 작업 내역

### 추가된 번역 키

| 키 | ko | en | ja |
|----|----|----|----|
| auth.login.title | 로그인 | Login | ログイン |

### 수정된 파일

- /src/locales/ko.json
- /src/locales/en.json
- /src/locales/ja.json
- /src/pages/Login.tsx
```

PLAN 파일이 없는 경우:

```text
/docs/current_sprint/planMMDDNN.md
```

위 형식으로 간단한 PLAN 파일을 생성합니다.

## 실행 순서

### Phase 1: 현황 파악

1. 로케일 디렉토리 구조 확인
   ```text
   /src/locales/
   ├── ko.json          # 한국어 (기본 언어)
   ├── en.json          # 영어
   ├── ja.json          # 일본어
   └── index.ts         # 로케일 로더
   ```
   > 플랫폼별 표준이 있는 경우 해당 표준을 따릅니다 (예: Swift의 .strings, Android의 strings.xml)

2. 지원 언어 목록 확인
3. 기본 언어 파일(ko.json)의 키 목록을 기준으로 설정
4. l10n 설정이 없으면 위 파일 구조로 새로 구성

### Phase 2: 누락 항목 식별

1. 하드코딩된 텍스트 검색
   - UI 텍스트 (버튼, 라벨, 제목, 설명)
   - 에러 메시지
   - placeholder, alt 텍스트
2. 신규 추가된 컴포넌트 확인
3. **언어 파일 간 키 불일치 검사** (기본 언어 기준)
   - 한 언어에만 존재하는 키
   - 누락된 키 목록화
4. 누락 항목 목록 작성

### Phase 3: 번역 키 추가

1. 번역 키 생성 (네이밍 규칙 준수)
   ```text
   <domain>.<page>.<element>.<description>

   예시:
   - common.button.save
   - common.button.cancel
   - auth.login.title
   - auth.login.error.invalidPassword
   - dashboard.chart.noData
   ```

2. **모든 언어 파일에 동일 키 추가** (누락 방지)
   ```json
   // ko.json
   {
     "common.button.save": "저장"
   }

   // en.json
   {
     "common.button.save": "Save"
   }

   // ja.json
   {
     "common.button.save": "保存"
   }
   ```

3. 컴포넌트에서 번역 키 사용으로 교체

### Phase 4: 검증

1. **키 일관성 검사**: 모든 언어 파일의 키가 동일한지 확인
   - 자동화 스크립트 활용 권장
2. 각 언어별 화면 확인
   - 번역이 정상적으로 표시되는지
3. 텍스트 잘림/오버플로우 확인
   - 긴 번역 텍스트 처리
4. 누락 항목 재확인
5. 빌드 및 테스트 실행

### Phase 5: 문서화

1. PLAN 파일에 l10n 작업 내역 기록
2. 추가된 번역 키 목록 정리
3. 수정된 파일 목록 기록

## 점검 대상

| 대상 | 예시 |
|------|------|
| UI 텍스트 | 버튼, 라벨, 제목, 설명 |
| 에러 메시지 | validation, API 에러 |
| placeholder | input, textarea |
| alt 텍스트 | 이미지 설명 |
| title/tooltip | 마우스오버 텍스트 |
| 확인/취소 | 다이얼로그 버튼 |

## 제외 대상

| 대상 | 이유 |
|------|------|
| 로그 메시지 | 개발자용, 번역 불필요 |
| 코드 주석 | 개발자용 |
| 테스트 코드 | 내부용 |
| 고유명사 | 브랜드명, 제품명 등 |

## 번역 키 네이밍 규칙

```text
<domain>.<page>.<element>.<description>

예시:
- common.button.save           # 공통 저장 버튼
- common.button.cancel         # 공통 취소 버튼
- auth.login.title             # 로그인 페이지 제목
- auth.login.error.invalidPassword  # 로그인 에러 메시지
- dashboard.chart.noData       # 대시보드 차트 데이터 없음
```

**네이밍 원칙:**
- 도메인부터 구체적인 요소까지 계층적 구성
- 소문자 사용
- 점(.)으로 구분
- 명확하고 일관된 이름 사용

## 언어 파일 형식

```json
// ko.json (기본 언어)
{
  "common.button.save": "저장",
  "common.button.cancel": "취소",
  "auth.login.title": "로그인",
  "auth.login.error.invalidPassword": "비밀번호가 올바르지 않습니다"
}

// en.json
{
  "common.button.save": "Save",
  "common.button.cancel": "Cancel",
  "auth.login.title": "Login",
  "auth.login.error.invalidPassword": "Invalid password"
}

// ja.json
{
  "common.button.save": "保存",
  "common.button.cancel": "キャンセル",
  "auth.login.title": "ログイン",
  "auth.login.error.invalidPassword": "パスワードが正しくありません"
}
```

## 주의사항

- **키 추가/삭제 시 모든 언어 파일 동시 수정** (필수)
- 기본 언어(ko.json)를 기준으로 다른 언어 파일 동기화
- 번역 키 삭제 전 사용처 확인
- 번역이 없는 경우 기본 언어 값 또는 키 자체를 fallback으로 표시
- 긴 텍스트는 UI 레이아웃 고려
- 플랫폼별 표준 형식 준수 (Swift .strings, Android strings.xml 등)
- 관련 문서: [CLAUDE.md](../../CLAUDE.md)
